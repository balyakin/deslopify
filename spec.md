# Deslopify: Python LLM Slop Linter

## Описание проекта
**Deslopify** — это расширяемый статический анализатор кода (линтер) для Python-проектов. Его главная цель — находить специфический "мусор" (slop), галлюцинации и стилистические ошибки, которые часто оставляют большие языковые модели (LLM) при генерации или рефакторинге кода. 

Примеры таких ошибок:
- Пустые функции или методы (состоящие только из `pass`).
- Блоки `try...except pass`.
- Публичные атрибуты классов, которые используются только внутри самого класса (должны быть приватными `_name`).

## Ключевые требования
1. **Совместимость:** Строго Python 3.5. Запрещено использование f-строк (используется `.format()`) и новых аннотаций типов.
2. **Глобальный запуск:** Утилита должна запускаться через system-wide alias (например, в macOS) из корня любого проверяемого (целевого) проекта.
3. **Игнорирование окружения:** При обходе целевого проекта линтер должен автоматически игнорировать служебные директории: `.venv`, `venv`, `.git`, `.idea`, `__pycache__` и т.д.
4. **Плагинная архитектура:** Каждая проверка (правило) изолирована в отдельном плагине со своей документацией.
5. **Двухпроходный анализ:** Поддержка как локальных проверок (в рамках одного файла), так и глобальных (требующих контекста всего проекта). Реализуется через фазу сбора данных и фазу финализации.

## Архитектура
Проект состоит из следующих основных модулей:

- `main.py` — CLI-интерфейс (точка входа). Определяет целевую директорию (`os.getcwd()`), настраивает пути и запускает Runner.
- `runner.py` — ядро линтера. Выполняет обход файловой системы целевого проекта, читает `.py` файлы, парсит их в AST (Abstract Syntax Tree) и прогоняет через все загруженные плагины.
- `plugin_manager.py` — отвечает за динамическое обнаружение и импорт плагинов из папки `plugins/`.
- `core/base_checker.py` — базовый класс `BaseChecker`, от которого наследуются все плагины. Расширяет `ast.NodeVisitor`.

## Структура директорий
```text
deslopify/
├── main.py                  
├── runner.py                
├── plugin_manager.py        
├── core/
│   ├── __init__.py
│   └── base_checker.py      
└── plugins/                 # Каталог с плагинами
    ├── empty_function/      # Пример плагина
    │   ├── __init__.py
    │   ├── checker.py       
    │   └── README.md        
    └── try_except_pass/
        ├── __init__.py
        ├── checker.py
        └── README.md
```

## Жизненный цикл проверки (Execution Flow)
1. **Инициализация:** `plugin_manager` загружает все классы чекеров из `plugins/`.
2. **Парсинг:** `runner` находит все валидные `.py` файлы в целевом проекте и преобразует их в AST-деревья.
3. **Phase 1: Process (Сбор данных):** Для каждого файла и каждого чекера вызывается метод `process_file(filename, tree)`. В этот момент чекер обходит AST файла. Локальные ошибки можно фиксировать сразу. Глобальные данные собираются во внутреннее состояние чекера.
4. **Phase 2: Finalize (Анализ контекста):** После обхода всех файлов у каждого чекера вызывается метод `finalize()`. Здесь чекер анализирует собранные глобальные данные всего проекта и фиксирует комплексные ошибки.
5. **Вывод:** Все найденные ошибки группируются и выводятся в консоль.

## Установка и использование (macOS / Linux)
Для удобного вызова линтера из любого каталога, добавьте следующий алиас в ваш файл конфигурации оболочки (например, `~/.zshrc` или `~/.bash_profile`):

```bash
alias deslopify="python3.5 /Users/evgenybalyakin/PycharmProjects/PythonProject/deslopify/main.py"
```

После этого обновите конфигурацию (`source ~/.zshrc`) и запускайте анализ командой:
```bash
deslopify
```
Утилита просканирует текущую директорию (целевой проект), используя свои плагины из директории установки.
