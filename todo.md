A) Исключения и подавление ошибок (1–20)
Bare except: наличие except: без указания типа исключения.
Слишком широкий catch: except Exception: (или шире) без объясняющего комментария/обоснования.
Ловля BaseException/SystemExit/KeyboardInterrupt: except BaseException или явная ловля SystemExit, KeyboardInterrupt в обычном коде.
DONE: Поглощение ошибки “в ноль”: except ...: pass.
Поглощение ошибки “в многоточие”: except ...: ... (Ellipsis как тело).
Поглощение через дефолт: except ...: return None/False/0/[]/{} без логирования/комментария.
Поглощение через print: except ...: print(...); return/continue (вместо нормального логирования/обработки).
except ... as e, но e не используется (LLM часто так пишет “по шаблону”).
Потеря traceback: внутри except делается raise e вместо raise.
Потеря контекста исключения: raise Exception(str(e)) / raise RuntimeError(...) без from e.
Слишком большой try‑блок: try покрывает >N операторов/строк (эвристика “обернули всё подряд”).
Один try — много причин падения: внутри try есть несколько “разнородных” операций (I/O + парсинг + бизнес‑логика) и один общий except.
Чрезмерная вложенность try/except: глубина вложенности try > N (например, >2).
Пустой finally: finally: pass или finally: ....
finally подавляет исключение: в finally есть return/break/continue (это может “съесть” исключение).
except с continue без обработки: except ...: continue без логирования/метки.
contextlib.suppress злоупотребление: suppress(Exception) или suppress вокруг большого блока кода.
Глобальное игнорирование warnings: warnings.filterwarnings("ignore") / simplefilter("ignore") на уровне модуля.
Логируем и делаем вид, что всё ок: logging.exception(...) + возврат “успешного” значения без явной деградации/сигнала.
assert в прод‑логике: assert используется для проверок входных данных/инвариантов в runtime‑коде (не в тестах).
B) Заглушки, псевдокод и артефакты генерации (21–45)
DONE: Пустая функция: тело функции — только pass.
Ellipsis как реализация: тело функции — только ... или raise NotImplementedError в обычном (не абстрактном) классе.
Пустой класс: class X: pass / ... без содержимого (не dataclass/не marker‑type) — подозрительно.
Пустой метод в неабстрактном классе: метод объявлен, но пустой (pass/...).
Фальш‑ветка: блоки if False: / if 0: / while False: (часто “оставлено на потом”).
TODO/FIXME/XXX/HACK в не‑тестовом коде.
TODO в docstring: docstring содержит “TODO”, “implement later”, “fill this in”, “placeholder”.
Подпись LLM в тексте: строки/комменты “As an AI language model…”, “Here is the code…”, “I can’t…” и т.п.
Явные placeholder‑имена: PLACEHOLDER, TBD, TODO, your_*_here, REPLACE_ME.
Placeholder‑секреты: API_KEY="YOUR_API_KEY" / “INSERT_TOKEN_HERE” и т.п. (или ключ в явном виде).
Placeholder‑URL по умолчанию: https://example.com, http://localhost, 127.0.0.1 как дефолт в библиотечном коде.
Placeholder‑аккаунты: "user@example.com", "admin", "password", "changeme" в коде.
Код выполняется при импорте: I/O/запросы/запуск логики на уровне модуля (не под if __name__ == "__main__":).
Демо‑функции без использования: большая “примерная” функция/блок, который нигде не вызывается.
“Сложная” функция всегда возвращает константу: например, всегда True или [] независимо от входа (по CFG).
Супер‑общие названия: функции process/handle/do_task/do_stuff/main_function без доменных слов (эвристика).
Низкая информативность имён: много foo/bar/baz/tmp/data/result в одном файле выше порога.
Похожесть docstring‑ов: docstring‑и у многих функций почти одинаковые (shingling/Levenshtein).
Комментарии “что делает строка”: комменты, которые буквально пересказывают код построчно (частый LLM‑паттерн).
Смешение языков: резкий микс RU/EN в комментариях/доках в одном модуле без причины (эвристика).
Дубликаты функций/классов: два блока кода почти идентичны (token similarity) — признак копи‑пасты генерации.
“Option A / Option B”: в файле оставлены альтернативные реализации, одна закомментирована/не используется.
Параметры “на будущее”: аргумент есть, но нигде не используется и не помечен _/unused_*.
Импорты “может пригодятся”: импорт есть, но нигде не используется + коммент в духе “for future use”.
main‑блок как “мини‑тест”: if __name__ == "__main__": содержит случайные print/ручные примеры вместо CLI/тестов (эвристика slop).
C) Импорты и зависимости (46–60)
Unused import: импортирован модуль/имя, но не используется (AST symbol table).
Unused from‑import: from x import y, но y не используется.
Дублирующиеся импорты: один и тот же модуль/имя импортируется несколько раз.
import *: from module import *.
Неоправданный импорт в горячей функции: импорт внутри функции, которая потенциально вызывается часто (и импорт не условный).
Опциональная зависимость “проглочена”: try: import ... except: pass без fallback/ошибки/флага доступности.
Тяжёлые либы ради мелочи: pandas/numpy для пары операций, когда хватает стандартной библиотеки (эвристика).
Теневая подмена stdlib: локальный модуль конфликтует с stdlib (json.py, typing.py, email.py) — можно ловить по путям/графу импортов.
Затирание builtins импортом/присваиванием: list = ..., dict = ..., type = ... (в т.ч. через импорт).
Разные алиасы для одного модуля: один и тот же модуль импортирован под разными именами в проекте/файле.
Циклические импорты: обнаружение cycle в графе импортов проекта.
Использование deprecated API (под твой целевой Python): поиск по известным deprecated‑символам/паттернам.
Несовместимость с target‑версией Python: использование фич/typing‑символов без guards для заявленной версии.
sys.path.append(...) для импортов: ручная правка путей импорта в прод‑коде (часто “починили генерацией”).
Платформенный импорт с пустой веткой: if sys.platform ...: import X else: pass.
D) Мёртвый код, неиспользуемые сущности, отладочные артефакты (61–80)
Unreachable после control‑flow: код после return/raise/break/continue.
Константные условия: if True: с else, if 1==1, if DEBUG где DEBUG — константа в модуле.
Мёртвый else‑шум: else после return во всех ветках if (code smell).
Assigned but unused vars (кроме _).
Выражение без эффекта: standalone‑expression без side‑effects (x + 1 строкой).
Невостребованные функции/классы: дефайны, которые нигде не вызываются/не импортируются (по графу вызовов проекта).
Невостребованные приватные хелперы: _helper нигде не используется (особенно в маленьких модулях).
Debug print в библиотеке: print() в модуле, который явно не CLI (эвристика по структуре проекта).
Следы дебаггера: breakpoint(), pdb.set_trace(), ipdb imports.
logging.basicConfig() в библиотеке (side‑effect на import).
Большие закомментированные блоки: >N строк кода в комментариях или triple‑quoted “комментарии”.
Чрезмерные подавления: много # noqa / # type: ignore (выше порога на файл/проект).
Пустые циклы: for ...: pass / while ...: pass.
Потенциально бесконечный цикл: while True: без гарантированного break/return/raise по CFG.
Сайд‑эффекты через list‑comprehension: [do(x) for x in xs] результат не используется.
Comprehension ради print: [print(x) for x in xs] и подобное.
Шаблонная ловля e без использования по проекту: много except ... as e: и e нигде не используется (метрика).
Перезапись переменных без использования старого значения: x = ...; x = ... подряд без чтения.
Дубли констант/настроек: одинаковые литералы/конфиги копипастятся в разных файлах (похожесть).
“Мёртвая конфигурация”: читаются env/args/keys, но дальше по data‑flow они нигде не влияют на поведение.
E) Частые баги/антипаттерны, которые LLM часто лепит (81–100)
DONE: Mutable default args: def f(x=[]/{} / set() / SomeObj()).
Сравнение с None через ==/!= вместо is/is not.
is для строк/чисел: x is "a" / x is 1.
Смешивание naive/aware datetime: операции/сравнения datetime без tzinfo vs с tzinfo.
datetime.now()/utcnow() без явной TZ‑политики в серверном/бизнес‑коде (эвристика по модулям).
“try вместо нормальной проверки” для dict: try: v=d[k] except: v=... (особенно с широким except).
random для секретов: генерация токенов/паролей через random.* вместо secrets.
hash() как постоянный идентификатор (не стабилен между запусками).
Файлы без context manager: f = open(...); ...; f.close() отсутствует/не гарантируется.
open() без encoding= в текстовом режиме (особенно для библиотек/CLI).
requests без timeout: requests.get/post/... без timeout=.
verify=False / отключение SSL‑проверки в requests/urllib + подавление warnings.
SQL через конкатенацию/f-string: f"SELECT ... {user_input}" вместо параметров.
subprocess с shell=True или команда строится строкой из данных (особенно из внешнего ввода).
eval/exec на не‑литерале: eval(x) где x не константная строка.
pickle load из недоверенного источника: pickle.loads/load на данных из файла/сети без trust‑границы.
yaml.load без SafeLoader (PyYAML): yaml.load(...) без Loader=yaml.SafeLoader.
Небезопасные временные файлы/права: tempfile.mktemp, chmod 777, запись в предсказуемое имя в /tmp.
Несогласованные return‑типы: в части веток возвращается значение, в части — None (и это не помечено Optional/не документировано).
Локальное затирание builtins: переменные/параметры id, type, list, dict, input, file, sum и т.п. в рабочих областях.
